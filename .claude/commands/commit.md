커밋을 진행한다. 아래 순서를 따른다.

## 분석 단계 (사용자 개입 없이 순서대로 진행)

1. `docs/git/COMMIT_CONVENTION.md`를 읽어 커밋 메시지 규칙을 확인한다.

2. `git status`와 `git diff HEAD`를 실행해 변경 사항을 파악한다.

3. `.py` 파일 변경이 포함된 경우에만 문서 동기화 분석을 수행한다. 문서·설정 파일만 변경된 경우 이 단계를 스킵한다.
   - `docs/` 디렉터리를 탐색해 변경된 코드와 관련된 문서를 찾는다.
   - 각 문서를 읽고 실제 코드와 불일치하는 항목(함수명, 경로, 설명 등)을 식별한다.
   - 업데이트가 필요한 문서와 변경 내용을 목록으로 정리해 둔다. **아직 수정하지 않는다.**

4. `PROGRESS.md`를 읽고 변경 사항 반영 필요 여부를 판단한다.
   - 반영이 필요하면 업데이트 내용을 정리해 둔다. **아직 수정하지 않는다.**
   - 완료된 작업은 `HISTORY.md`로 이동하고 `PROGRESS.md`는 다음 작업 기준으로 정리하는 내용을 준비한다.

5. COMMIT_CONVENTION.md 규칙에 맞는 커밋 메시지 초안을 작성해 둔다.

## 통합 제안 (1회 승인)

6. 아래 내용을 한 번에 사용자에게 보여주고 승인을 받는다.

   ```
   [문서 동기화] (해당 없으면 "없음")
   - 수정할 문서 및 변경 내용 목록

   [PROGRESS.md] (변경 없으면 "없음")
   - 변경할 내용 요약

   [커밋 메시지]
   type: 제목

   본문 (있는 경우)
   ```

## 실행 단계 (승인 후)

7. 승인되면 아래를 순서대로 실행한다.
   - 3단계에서 식별한 문서들을 업데이트한다.
   - 4단계에서 준비한 내용으로 `PROGRESS.md`와 `HISTORY.md`를 업데이트한다.
   - 변경된 파일을 스테이징하고 커밋을 실행한다.

8. 커밋 완료 후 push 여부를 사용자에게 확인한다. 승인 시 현재 브랜치 기준으로 `git push`를 실행한다.
